PHASE 3 · 에이전트 심화
👋 오늘의 한마디

멀티 에이전트 시스템의 핵심은 "누가 지휘하는가"에 있습니다. 오케스트레이터가 워커 에이전트들을 효율적으로 조율하려면, 명확한 메시지 포맷과 일관된 통신 프로토콜이 필수입니다. 오늘은 실전 프로젝트에 바로 적용할 수 있는 두 가지 패턴을 다룹니다.

🎯 오늘의 팁 2선
#143 ⭐ 고급 멀티 에이전트 아키텍처

고급 오케스트레이터 — 오케스트레이터-워커 패턴

오케스트레이터-워커 패턴은 하나의 중앙 에이전트(오케스트레이터)가 여러 전문가 에이전트(워커)의 작업을 조율하는 구조입니다. 오케스트레이터는 사용자 요청을 분석해 어떤 워커를 호출할지 결정하고, 각 워커의 결과를 통합하여 최종 답변을 생성합니다.

핵심 책임 분리: 오케스트레이터는 "관리자" 역할로, 복잡한 로직은 각 워커에게 위임합니다. 이렇게 하면 확장성과 유지보수성이 크게 향상됩니다. 예를 들어, 새로운 워커 기능이 필요하면 오케스트레이터 수정 없이 워커만 추가하면 됩니다.

💬 패턴 구조
사용자 입력 ↓ 오케스트레이터 ├─ 요청 분석 (어떤 워커 필요?) ├─ 워커 선택 (논리적 판단) └─ 결과 통합 ↓ 워커 1 (마케팅) ← 특정 작업 수행 워커 2 (기술) ← 특정 작업 수행 워커 3 (재무) ← 특정 작업 수행 ↓ 최종 답변 생성

#144 ⭐ 고급 에이전트 통신

고급 에이전트 통신 — 메시지 포맷과 프로토콜

에이전트 간 통신의 품질이 곧 시스템의 품질을 결정합니다. 구조화된 메시지 포맷을 사용하면 에이전트들이 서로 정확히 이해하고, 오류를 줄일 수 있습니다. JSON 기반의 표준 프로토콜을 권장합니다.

필수 요소: 각 메시지는 (1) 요청 ID (추적), (2) 작업 타입 (어떤 일), (3) 파라미터 (필요한 정보), (4) 우선순위 (처리 순서)를 포함해야 합니다. 이렇게 하면 비동기 처리, 재시도, 로깅이 모두 용이해집니다.

💬 메시지 포맷
{ "request_id": "req-2026-0616-001", "task_type": "market_analysis", "priority": "high", "parameters": { "industry": "반려동물용품", "region": "한국", "time_range": "2024-2026" }, "callback_url": "orchestrator/callback", "timeout_ms": 30000 }

💡 오늘의 인사이트

오케스트레이터-워커 패턴은 단순해 보이지만, 실전에서는 세 가지 도전이 있습니다: (1) 각 워커의 응답 시간이 다름 — 타임아웃 관리 필수, (2) 부분 실패 처리 — 한 워커 오류가 전체를 막으면 안 됨, (3) 상태 추적 — 누가 뭘 하는 중인지 파악해야 함. 이 세 가지를 해결하면 견고한 멀티 에이전트 시스템을 만들 수 있습니다.

📖 오늘의 용어
오케스트레이터
Orchestrator
멀티 에이전트 시스템에서 다른 에이전트들의 작업을 조율하고 관리하는 중앙 에이전트. 사용자 요청을 분석하여 어떤 워커 에이전트를 언제 호출할지 결정합니다.
워커 에이전트
Worker Agent
오케스트레이터의 지휘를 받아 특정 작업을 전담하는 전문 에이전트. 마케팅, 기술, 재무 등 특정 도메인에 특화되어 있으며, 할당받은 작업만 수행합니다.
