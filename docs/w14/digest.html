#131 🌱 입문 에이전트 기초

AI 에이전트란 무엇인가

에이전트(Agent)는 "주어진 목표를 달성하기 위해 환경과 상호작용하는 자율적 시스템"입니다. 단순 챗봇과의 가장 큰 차이는 생각하고 행동하는 루프를 반복한다는 점입니다.

예를 들어, "내 캘린더에서 다음 주 미팅 시간을 찾아 회의실을 예약해줘"라는 명령을 받으면:

1️⃣ 목표 이해: "회의실 예약"이 진짜 목표인가? 아니면 그 안에 캘린더 확인, 참석자 통보 등이 있는가?
2️⃣ 도구 선택: "캘린더 조회 도구" → "회의실 예약 도구" → "이메일 도구" 순서대로 사용
3️⃣ 실행과 피드백: 각 단계의 결과를 확인하고, 문제가 나면 다시 계획 수정
4️⃣ 최종 검증: "예약 완료했고, 참석자들에게 알림도 보냈습니다"라고 보고

🔄 에이전트 vs 챗봇
챗봇: "캘린더를 확인하려면 Google Calendar API를 연결해야 합니다. 다음은 연결 방법입니다..." (답만 함) 에이전트: 1. 캘린더 조회 도구 실행 → 회의 데이터 수집 2. 빈 시간 분석 → "다음 주 화요일 2PM이 최적" 3. 회의실 예약 도구 실행 → 예약 완료 4. 이메일 발송 도구 실행 → 참석자 알림 5. "모든 작업 완료. 다음 주 화요일 2PM, B-101호실" (직접 행동)

#132 🔵 중급 에이전트 아키텍처

에이전트의 4대 구성요소

모든 에이전트는 네 가지 핵심 요소로 이루어져 있습니다. 이것을 이해하면 어떤 에이전트든 설계할 수 있습니다:

1. 뇌(LLM, Large Language Model)
Claude, GPT-4 같은 거대 언어 모델. 상황을 분석하고 다음 행동을 결정합니다.

2. 도구(Tools)
에이전트가 실제 행동을 취할 수 있게 해주는 함수나 API. 예: 이메일 발송, 데이터베이스 조회, 웹 검색.

3. 메모리(Memory)
과거 상호작용과 학습 결과를 기억. 같은 실수를 반복하지 않고, 맥락을 유지합니다.

4. 피드백 루프(Feedback Loop)
행동 결과를 확인하고, 성공 여부를 판단한 뒤 다음 행동을 조정합니다.

┌─────────────────────────────────┐ │ 목표: "월드컵 경기 일정 알기" │ └─────────────────┬───────────────┘ │ ┌─────────▼─────────┐ │ 뇌(Claude) │ │ "웹 검색이 필요함" │ └─────────┬─────────┘ │ ┌─────────▼──────────┐ │ 도구: 웹 검색 │ │ 결과: JSON 데이터 │ └─────────┬──────────┘ │ ┌─────────▼──────────────┐ │ 피드백 루프: 데이터 검증 │ │ "정보 완성? Y/N" │ └─────────┬──────────────┘ │ ┌─────────▼──────────┐ │ 메모리: 저장 │ │ 다음 요청에 사용 │ └─────────┬──────────┘ │ ┌─────────▼──────────┐ │ 최종 답변 반환 │ └────────────────────┘

#133 🔵 중급 도구 설계

도구 사용(Tool Use) 이해하기

Tool Use는 에이전트가 자신의 능력을 확장하는 방식입니다. 에이전트는 LLM일 뿐이므로 직접 실행할 수 없는 작업을 도구를 통해 수행합니다.

핵심은 LLM이 "이 일은 도구가 필요하다"를 판단하고, 올바른 도구와 인자를 선택하는 것입니다.

Tool Use의 사이클:
1️⃣ 사용자 요청 분석
2️⃣ 필요한 도구 결정
3️⃣ 도구에 전달할 인자 준비
4️⃣ 도구 실행
5️⃣ 결과 분석 및 다음 도구 판단
6️⃣ 결과 통합 및 최종 답변

🛠️ Tool Use 예시
사용자: "런던의 오늘 날씨와 내일 날씨를 비교해서, 우산이 필요한지 알려줘" 에이전트의 사고 과정: 1. "날씨 정보가 필요하다" → 날씨 API 도구 선택 2. 도구에 {"city":"London", "date":"today"} 전달 3. 결과: {"temp":12°C, "condition":"rainy"} 4. 다시 {"city":"London", "date":"tomorrow"} 전달 5. 결과: {"temp":14°C, "condition":"sunny"} 6. 분석: "오늘은 비, 내일은 맑음" 7. 최종 답변: "오늘은 우산이 필수, 내일은 불필요"

#134 ⚡ 고급 에이전트 루프

에이전트 루프 설계 패턴

에이전트의 핵심은 think → act → observe 루프를 반복하는 것입니다. 이 루프를 얼마나 잘 설계하느냐가 에이전트의 성공을 좌우합니다.

세 가지 주요 루프 패턴:

패턴 1: ReAct (Reasoning + Acting)
LLM이 추론과 행동을 번갈아가며 수행. 가장 기본이면서도 효과적.

패턴 2: Plan-Then-Act
먼저 전체 계획을 세우고, 그 다음 순차적으로 실행. 복잡한 다단계 작업에 유리.

패턴 3: Hierarchical Loop
상위 에이전트가 큰 그림을 결정하고, 하위 에이전트가 각 단계를 실행. 매우 복잡한 시스템에 사용.

🔄 ReAct 패턴 예시
Iteration 1 - THINK: "사용자가 'Q1 실적' 데이터를 요청했다. 데이터베이스 조회 도구가 필요하다." Iteration 1 - ACT: query_db(sql="SELECT * FROM Q1_results") Iteration 1 - OBSERVE: {"revenue":1000000, "profit":250000, ...} Iteration 2 - THINK: "데이터를 받았다. 이제 분석이 필요하다. 작년과 비교하려면 Q1_last_year도 필요하다." Iteration 2 - ACT: query_db(sql="SELECT * FROM Q1_results_2024") Iteration 2 - OBSERVE: {"revenue":900000, "profit":200000, ...} Iteration 3 - THINK: "두 데이터를 비교할 수 있다. 최종 답변을 구성하자." 최종 답변: "Q1 매출: 1,000만원 (작년 대비 +11.1%) Q1 이익: 250만원 (작년 대비 +25%)"

#135 ⚡ 고급 에이전트 안전성

에이전트의 실패 모드와 안전장치

자율적인 에이전트일수록 통제 불가능한 행동의 위험이 커집니다. 무한 루프, 도구 오남용, 할루시네이션(거짓 정보 생성) 등이 주요 실패 모드입니다.

주요 실패 모드 4가지:

1️⃣ 무한 루프(Infinite Loop): 에이전트가 같은 도구를 계속 호출해 진행이 안 됨.
→ 해결: 최대 반복 횟수 제한, 도구 호출 이력 추적

2️⃣ 할루시네이션(Hallucination): 존재하지 않는 도구나 데이터를 사용한다고 주장.
→ 해결: 사용 가능한 도구를 명확히 명시, 결과 검증

3️⃣ 잘못된 도구 선택: 이메일 도구로 데이터베이스를 쿼리하려 시도.
→ 해결: 도구 설명을 상세히, 테스트 케이스로 검증

4️⃣ 권한 이상(Privilege Escalation): 본래 권한 이상으로 행동 수행.
→ 해결: 도구별 권한 체크, 감시 로깅

🛡️ 안전장치 구현 예
MAX_ITERATIONS = 10 # 최대 10번 반복 history = [] # 도구 호출 이력 for iteration in range(MAX_ITERATIONS): # 도구 호출 tool_result = call_tool(...) history.append(tool_result) # 같은 도구를 3번 연속 호출했으면 중단 if len(history) >= 3: if history[-1] == history[-2] == history[-3]: raise LoopDetected("무한 루프 감지됨") # 도구가 존재하는지 확인 if tool_name not in AVAILABLE_TOOLS: raise ToolNotFound(f"{tool_name}은 없음") # 도구 실행 권한 확인 if not user.has_permission(tool_name): raise PermissionDenied(f"권한 없음: {tool_name}")

#136 🔵 중급 LLM 통합

Claude에서 MCP 서버 활용하기

MCP(Model Context Protocol)는 Anthropic이 제안한 표준화된 도구 연결 방식입니다. Claude 에이전트가 다양한 외부 시스템에 쉽게 연결되도록 합니다.

MCP를 사용하면 별도의 복잡한 통합 로직 없이 플러그앤플레이처럼 도구를 추가할 수 있습니다.

MCP의 세 가지 핵심 역할:

1️⃣ Tool Definition(도구 정의): 도구가 뭔지, 어떤 인자를 받는지 명시
2️⃣ Resource Access(리소스 접근): 파일, 데이터베이스 등 접근
3️⃣ Prompt Support(프롬프트 지원): 도구 사용 방법을 명확하게 설명

🔌 MCP로 도구 정의
{ "tools": [ { "name": "get_weather", "description": "주어진 도시의 날씨 정보 조회", "inputSchema": { "type": "object", "properties": { "city": { "type": "string", "description": "도시명" }, "date": { "type": "string", "description": "조회 날짜 (YYYY-MM-DD)" } }, "required": ["city"] } } ] }

#137 🔵 중급 검색·생성 통합

검색 보강 생성(RAG) 기초

RAG(Retrieval-Augmented Generation)는 "검색 → 생성" 구조로 에이전트의 답변 정확도를 크게 높입니다. 단순한 LLM의 고정 지식에만 의존하지 않고, 최신 정보를 동적으로 가져옵니다.

RAG의 세 단계:

1️⃣ Retrieval(검색): 사용자 질문과 관련된 문서/데이터 검색
2️⃣ Context 준비: 검색된 정보를 프롬프트에 포함
3️⃣ Generation(생성): 검색 결과를 바탕으로 답변 생성

에이전트 관점에서 RAG는 "검색" 도구의 고도화 버전입니다. 단순 웹 검색 대신, 회사 내부 문서나 데이터베이스에서도 정보를 가져올 수 있습니다.

🔍 RAG 사이클
사용자 질문: "우리 회사 2024년 경영 방침은 뭐야?" 1️⃣ RETRIEVAL: 내부 wiki 검색 + 임베딩 유사도 계산 → "2024_company_strategy.pdf" 검색됨 2️⃣ CONTEXT 준비: 문서에서 "경영 방침" 섹션 추출 → "Digital-First, Cost Optimization, ..." 3️⃣ GENERATION: 프롬프트: "다음 문서를 바탕으로 경영 방침을 설명: [검색된 내용] 질문: 우리 회사 2024년 경영 방침은?" 최종 답변: "2024년 우리 회사의 경영 방침은 다음 세 가지입니다: 1. Digital-First Transformation 2. Cost Optimization 3. ..."

#138 ⚡ 고급 함수 호출

함수 호출(Function Calling) 실전

Function Calling은 도구 사용의 기술적 구현입니다. LLM이 "이 함수를 이 인자로 호출하라"고 명시적으로 말하면, 시스템이 그 함수를 실행하는 방식입니다.

핵심은 LLM의 출력을 구조화하는 것입니다. JSON 형식으로 도구 이름과 인자를 명시하면, 프로그램이 자동으로 파싱해 실행할 수 있습니다.

Function Calling의 실전 흐름:
1️⃣ 사용자 질문과 함께 "사용 가능한 함수 목록" 전달
2️⃣ LLM이 어떤 함수를 써야 할지 결정
3️⃣ LLM이 함수명과 인자를 JSON으로 반환
4️⃣ 시스템이 JSON을 파싱해 함수 실행
5️⃣ 실행 결과를 다시 LLM에 전달
6️⃣ LLM이 결과를 해석해 최종 답변 생성

📞 Function Calling 예시
사용자 요청: "서울의 내일 날씨 알려줘" 시스템이 LLM에 전달: available_functions: [ { "name": "get_weather", "parameters": { "city": "string", "date": "string (YYYY-MM-DD)" } } ] LLM의 응답: { "type": "function_call", "function": { "name": "get_weather", "arguments": { "city": "Seoul", "date": "2026-04-08" } } } 시스템이 함수 실행: result = get_weather(city="Seoul", date="2026-04-08") → {"temp": 15, "condition": "rainy"} LLM이 최종 답변: "서울의 내일 날씨는 빗소식입니다. 기온은 15°C 정도로 쌀쌀하니 우산을 챙기세요."

#139 ⚡ 고급 에이전트 메모리

에이전트 메모리 설계

메모리(Memory)는 에이전트가 과거에서 배우고, 문맥을 유지하는 능력입니다. 메모리 없이는 매번 같은 실수를 반복하고, 대화의 맥락을 잃습니다.

세 가지 메모리 타입:

1️⃣ Short-term Memory(단기 메모리): 현재 대화 내에서의 문맥. 최근 메시지들만 저장.
→ 사용: 매 질문에 최근 5개 메시지 포함

2️⃣ Long-term Memory(장기 메모리): 여러 대화에 걸친 사실과 패턴. 데이터베이스에 저장.
→ 사용: "사용자는 마케팅 담당자다"같은 정보

3️⃣ Episodic Memory(에피소드 메모리): 과거 상호작용의 구체적 기록. "아, 지난주 이 에러를 해결했었다"는 식의 기억.
→ 사용: 해결된 문제 목록, 실패 사례 분석

🧠 메모리 저장 예시
# 단기 메모리 (현재 세션) conversation_history = [ {"role": "user", "content": "분기 실적 분석해줘"}, {"role": "assistant", "content": "..."}, {"role": "user", "content": "작년과 비교하면?"} ] # 장기 메모리 (사용자 프로필) user_profile = { "name": "박마케팅", "role": "Marketing Manager", "department": "B2B Marketing", "preferences": ["data-driven", "brief_summary"] } # 에피소드 메모리 (문제 해결 이력) solved_issues = [ { "date": "2026-04-01", "issue": "CSV 임포트 실패", "solution": "인코딩을 UTF-8로 변경", "tags": ["data", "import"] } ] # LLM이 새로운 CSV 문제 받으면: if "CSV" in current_query and "import" in current_query: → solved_issues에서 유사한 해결책 검색 → "작년에 이런 문제 해결했어요!"라고 제안

#140 ⚡ 고급 평가 및 최적화

에이전트 성능 평가 방법론

에이전트의 성능을 어떻게 측정할까요? 단순한 챗봇처럼 "사용자 만족도"만으로는 부족합니다. 에이전트는 실제 작업을 수행하므로, 목표 달성도를 객관적으로 평가해야 합니다.

네 가지 핵심 지표(KPI):

1️⃣ Success Rate(성공률): 에이전트가 목표를 달성한 비율. 100회 중 95회 성공 = 95%
→ 목표: 90% 이상

2️⃣ Cost Efficiency(비용 효율): 작업당 토큰 사용량, API 호출 수. 에이전트가 불필요한 도구를 과다 호출하는지 확인.
→ 목표: 평균 이하 토큰 사용

3️⃣ Latency(처리 속도): 요청부터 답변까지 걸린 시간. 실시간 응답이 필요한 시스템에서 중요.
→ 목표: 평균 3초 이내

4️⃣ Human Intervention Rate(인간 개입률): 에이전트가 스스로 해결하지 못해 인간이 개입해야 했던 비율.
→ 목표: 5% 이하

📊 평가 프레임워크
평가 대상: "일정 자동 조정 에이전트" 테스트 케이스 100개 실행: ✅ 성공 케이스: 92개 - 올바른 일정 변경: 85개 - 참석자 통보까지 완료: 92개 → Success Rate: 92% 💰 비용 효율: - 평균 토큰 사용: 1,200 토큰/요청 - 평균 API 호출: 2.5회 → 예상 월 비용: $15 (100개 요청/일 기준) ⚡ 처리 속도: - 평균 시간: 2.3초 - 최대 시간: 5.1초 → Latency: Good 👤 인간 개입률: - 재일정이 불가능한 경우: 8개 → Intervention Rate: 8% 개선 권고: 1. 일정 충돌 해결 로직 강화 → Success Rate 95%로 2. 불필요한 정보 조회 제거 → Cost 10% 절감 3. 병렬 처리 구현 → Latency 1.5초로

💡 큰 그림: W14를 마치며

지난 5일 동안 에이전트의 모든 층을 살펴봤습니다. 개념(무엇인가) → 설계(어떻게 만드나) → 실행(도구 사용) → 안전(어떻게 보호하나) → 최적화(어떻게 평가하나)라는 전체 흐름입니다.

에이전트를 만든다는 것은 단순히 "더 똑똑한 챗봇"을 만드는 게 아닙니다. "스스로 생각하고, 행동하고, 실패에서 배우는 자율 시스템"을 창조하는 것입니다. W15부터는 이 개념들을 직접 코드로 구현하는 실전 주간이 시작됩니다. 당신은 이제 에이전트의 언어를 읽을 준비가 되었습니다.

📖 이번 주 핵심 용어
ReAct
Reasoning + Acting
LLM이 추론(Reasoning)과 행동(Acting)을 반복하면서 문제를 해결하는 패턴. 투명하고 추적 가능한 의사결정 과정을 제공합니다.
Tool Use
Tool Use / Function Calling
에이전트가 LLM 자체의 능력을 넘어서 외부 도구(API, 함수, 데이터베이스)를 호출하고 활용하는 기능.
RAG
Retrieval-Augmented Generation
외부 정보 소스에서 관련 데이터를 검색한 후, 그 정보를 바탕으로 텍스트를 생성하는 기법. LLM의 고정 지식의 한계를 극복합니다.
MCP
Model Context Protocol
Anthropic이 제안한 표준화된 도구 연결 프로토콜. Claude와 다양한 외부 시스템을 쉽게 통합할 수 있습니다.
할루시네이션
Hallucination
LLM이 존재하지 않는 정보나 거짓된 사실을 마치 참인 것처럼 생성하는 현상. 에이전트의 주요 실패 모드 중 하나.
