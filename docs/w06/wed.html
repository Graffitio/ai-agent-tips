👋 오늘의 한마디

프롬프트를 썼는데 결과가 마음에 들지 않을 때, 무작정 다시 쓰는 것보다 "뭐가 빠졌나?"를 체계적으로 점검하는 것이 훨씬 효율적입니다. 오늘은 디버깅 체크리스트와 이를 실행하는 프레임워크인 RACE를 소개합니다.

🎯 오늘의 팁 2선
#055 ⭐ 고급 프롬프트 디버깅

고급 디버깅 프롬프트 디버깅 체크리스트

AI의 답변이 만족스럽지 않을 때, 다음 체크리스트를 순서대로 확인하세요. 대부분의 문제는 이 다섯 가지 중 하나입니다.

✅ 프롬프트 디버깅 5단계
1️⃣ 역할(Role) 확인 → AI가 어떤 입장에서 답해야 하는지 명시했는가? 예: "마케팅 전문가로서", "초등학생도 이해할 수 있게" 2️⃣ 맥락(Context) 확인 → 충분한 배경 정보를 제공했는가? 예: 산업, 타겟, 현재 상황, 제약 조건 3️⃣ 출력 형식(Format) 확인 → 원하는 결과 형식을 명시했는가? 예: "JSON 형식으로", "표 형태로", "3줄 요약" 4️⃣ 예시(Examples) 추가 → 모범 답변을 하나라도 보여줬는가? 예: Before/After, 좋은 예시 1개 이상 5️⃣ 명확성(Clarity) 점검 → 모호한 표현이 있는가? 예: "좀 더", "충분히" → "구체적으로 수정"

이 체크리스트를 훑으면서 빠진 부분에 정보를 하나씩 추가하세요. 보통 3개 이상 추가하면 결과 품질이 크게 달라집니다.

#056 💙 중급 프레임워크

중급 RACE RACE 프레임워크

좋은 프롬프트의 필수 요소 4가지를 담은 프레임워크가 RACE입니다. 이 순서대로 프롬프트를 구성하면 AI의 답변 정확도가 비약적으로 올라갑니다.

🎯 RACE 프레임워크 구조
R (Role) - 역할 정의 ↳ AI가 취할 입장, 전문성, 톤앤매너 A (Action) - 수행할 작업 ↳ 구체적으로 무엇을 해야 하는가? C (Context) - 맥락 제공 ↳ 배경, 상황, 제약, 대상 정보 E (Expected output) - 기대 결과 정의 ↳ 형식, 길이, 스타일, 예시

실제 예시로 보겠습니다:

💬 RACE 프레임워크 실제 사용
R: 넌 경험 많은 UX 디자이너야. 유저 중심의 직관적인 설명을 한다. A: 앱의 온보딩 플로우를 3단계로 재설계해줘. C: 우리 앱은 AI 일정 관리 도구고, 타겟은 직장인 25-40세. 현재 온보딩은 기능 설명 중심인데, 너무 길어서 이탈률이 높아. E: 각 단계별로 스크린 설명, 유저 액션, 목표를 표 형태로 정리해줘. 왜 이렇게 설계했는지 1줄 근거도.

RACE의 각 요소가 명확할수록, AI의 답변도 더 정확하고 깊어집니다. 처음엔 시간이 걸리지만, 한두 번 연습하면 습관이 됩니다.

💡 오늘의 인사이트

"좋은 프롬프트"는 타고나는 것이 아니라 "체계적으로 만드는 것"입니다. 오늘 배운 디버깅 체크리스트와 RACE 프레임워크는 이를 위한 도구일 뿐입니다. 이제부터 AI에게 질문할 때마다 "내 프롬프트에 R, A, C, E가 모두 있나?"를 확인하세요. 이것만으로도 당신의 프롬프트 실력은 한 단계 업그레이드됩니다.

📖 오늘의 용어
프롬프트 디버깅
Prompt Debugging
AI의 답변이 예상과 다를 때, 프롬프트의 어느 부분이 부족하거나 모호한지 체계적으로 찾아내고 수정하는 과정입니다.
RACE 프레임워크
RACE Framework
Role(역할), Action(작업), Context(맥락), Expected output(기대 결과)의 네 가지 요소로 프롬프트를 구조화하는 방법론. 프롬프트의 완성도와 일관성을 높입니다.
